Die Philosophie hinter Handmade Hero

Handmade Hero ist nicht nur ein Tutorial — es ist eine komplette live-codierte Game-Engine von Casey Muratori. Das Projekt unterscheidet sich fundamental von fast allen anderen Lernressourcen, weil es keine Abstraktionen verwendet, die man nicht selbst verstanden hat.

🎯 Das Kernprinzip

Wenn du Unity oder Unreal verwendest, programmierst du gegen APIs, deren Implementierung du nicht kennst. Bei Handmade Hero schreibst du die API selbst. Du entscheidest, wie Memory allokiert wird, wie der Renderer arbeitet, wie Audio gemischt wird. Das Ergebnis ist ein tiefes, intuitives Verständnis für Computer.

Die Serie folgt keinem starren Lehrplan. Casey entwickelt die Engine explorationsbasiert — er trifft Design-Entscheidungen live, macht Fehler, korrigiert sie und erklärt dabei sein Denken. Das macht das Material authentisch und unglaublich wertvoll für jeden, der verstehen will, wie erfahrene Programmierer wirklich arbeiten.

  • Memory Management verstehen und Arenas, Pools und Custom Allocator bauen
  • Einen eigenen Software-Renderer und OpenGL-Renderer implementieren
  • Multithreading mit Lock-freien Queues und Semaphoren beherrschen

Phase 01 Tag 001 – 025

Windows-Plattform & Grundlagen

Bevor es ein Spiel gibt, brauchen wir eine stabile Plattform. In dieser Phase lernst du, wie man unter Windows ein Fenster öffnet, Sound ausgibt, Eingaben liest und einen flüssigen Backbuffer für Animationen aufbaut. Casey beginnt hier buchstäblich bei Null.

Tag 001–002: Das Win32-Fenster von Grund auf

Casey beginnt nicht mit einem Framework oder einer Engine. Er beginnt mit einer leeren C-Datei und der Frage: "Wie öffnet man unter Windows überhaupt ein Fenster?" Das ist bewusst so gewählt, denn das Verständnis der Grundlagen ist das Fundament für alles, was folgt.

Warum Win32 und nicht SDL oder GLFW?

Diese Frage stellt sich jeder Zuschauer in den ersten Minuten. Caseys Antwort ist klar: SDL und GLFW sind Abstraktionen. Sie verstecken die Windows-API hinter einer schönen Fassade. Wenn du sie verwendest, weißt du nicht, was im Hintergrund passiert. Wenn später ein Bug auftritt — ein Fenster, das sich nicht richtig verhält, ein VSync-Problem, ein Input-Lag — stehst du da, weil du die darunterliegende Schicht nicht verstehst.

Win32 ist hässlich, alt und voller historischer Ballast. Aber es ist die Wahrheit. Wenn du Win32 verstehst, verstehst du, wie Windows wirklich funktioniert. Und das Wissen ist universell: Egal ob du später SDL, GLFW, Qt oder eine eigene Engine baust — du weißt, was diese Bibliotheken für dich erledigen.

Schritt 1: Die minimale Windows-Anwendung erstellen

Öffne eine neue Datei win32_handmade.cpp. Jede Windows-Anwendung beginnt nicht mit main(), sondern mit WinMain(). Das ist der Einstiegspunkt, den Windows erwartet.

Was du genau tun sollst: Tippe den folgenden Code ab. Versuche nicht, ihn zu kopieren und zu ignorieren. Lies jede Zeile und verstehe sie. Das ist dein erstes Treffen mit der Windows-API.

win32_handmade.cpp — Das absolute Minimum
#include <windows.h>

LRESULT CALLBACK
MainWindowCallback(HWND Window, UINT Message, WPARAM WParam, LPARAM LParam)
{
    LRESULT Result = 0;
    
    switch(Message)
    {
        case WM_SIZE:
        {
            OutputDebugStringA("WM_SIZE\n");
        } break;
        
        case WM_DESTROY:
        {
            OutputDebugStringA("WM_DESTROY\n");
        } break;
        
        case WM_CLOSE:
        {
            OutputDebugStringA("WM_CLOSE\n");
        } break;
        
        case WM_ACTIVATEAPP:
        {
            OutputDebugStringA("WM_ACTIVATEAPP\n");
        } break;
        
        default:
        {
            Result = DefWindowProcA(Window, Message, WParam, LParam);
        } break;
    }
    
    return(Result);
}

int WinMain(HINSTANCE Instance, HINSTANCE PrevInstance, 
            LPSTR CommandLine, int ShowCode)
{
    WNDCLASSA WindowClass = {};
    WindowClass.lpfnWndProc = MainWindowCallback;
    WindowClass.hInstance = Instance;
    WindowClass.lpszClassName = "HandmadeHeroWindowClass";
    
    if(RegisterClassA(&WindowClass))
    {
        HWND WindowHandle = CreateWindowExA(
            0,
            WindowClass.lpszClassName,
            "Handmade Hero",
            WS_OVERLAPPEDWINDOW | WS_VISIBLE,
            CW_USEDEFAULT,
            CW_USEDEFAULT,
            CW_USEDEFAULT,
            CW_USEDEFAULT,
            0,
            0,
            Instance,
            0);
            
        if(WindowHandle)
        {
            for(;;)
            {
                MSG Message;
                BOOL MessageResult = GetMessage(&Message, 0, 0, 0);
                if(MessageResult > 0)
                {
                    TranslateMessage(&Message);
                    DispatchMessage(&Message);
                }
                else
                {
                    break;
                }
            }
        }
        else
        {
            // TODO: Logging
        }
    }
    else
    {
        // TODO: Logging
    }
    
    return(0);
}

Zeile für Zeile erklärt

WNDCLASS: Bevor Windows ein Fenster erstellen kann, musst du ihm sagen, wie dieses Fenster sich verhalten soll. Das geschieht über eine Window Class — eine Struktur, die die Eigenschaften einer Fensterkategorie beschreibt. Wichtigster Punkt: lpfnWndProc, die Window Procedure. Das ist die Funktion, die Windows jedes Mal aufruft, wenn etwas mit dem Fenster passiert.

RegisterClass: Die Window Class muss beim Betriebssystem registriert werden. Erst danach kannst du Fenster dieser Klasse erstellen.

CreateWindowEx: Hier passiert die eigentliche Fenstererstellung. WS_OVERLAPPEDWINDOW bedeutet: Das Fenster hat einen Rahmen, eine Titelleiste, Minimize-/Maximize-Buttons und kann in der Größe verändert werden. WS_VISIBLE sorgt dafür, dass es sofort sichtbar ist.

GetMessage: Dies ist die Message Loop — das Herz jeder Windows-Anwendung. Windows kommuniziert mit deinem Programm über Nachrichten (Messages). Wenn der Benutzer die Maus bewegt, eine Taste drückt oder das Fenster schließt, sendet Windows eine Message an dein Fenster. GetMessage holt diese Message aus der Warteschlange.

⚠️ Das Problem mit GetMessage

GetMessage ist blockierend. Wenn keine Messages vorhanden sind, hält es dein gesamtes Programm an. Für eine normale Desktop-Anwendung ist das perfekt — warum CPU-Zeit verschwenden, wenn nichts passiert? Aber für ein Spiel ist das tödlich. Ein Spiel muss kontinuierlich laufen: Es muss Audio ausgeben, Animationen berechnen, Physik simulieren — auch wenn der Spieler gerade stillhält und keine Taste drückt.

Die Lösung: PeekMessage

In Tag 002 ersetzt Casey GetMessage durch PeekMessage. Der entscheidende Unterschied: PeekMessage gibt sofort zurück, auch wenn keine Message vorhanden ist. Das ermöglicht uns, unsere eigene Schleife zu behalten.

Die Game Loop mit PeekMessage
Running = true;
while(Running)
{
    MSG Message;
    while(PeekMessage(&Message, 0, 0, 0, PM_REMOVE))
    {
        if(Message.message == WM_QUIT)
        {
            Running = false;
        }
        
        TranslateMessage(&Message);
        DispatchMessage(&Message);
    }
    
    // Hier kommt später unser Spiel-Code hin!
}

Beachte die verschachtelten Schleifen: Die äußere while(Running) ist unsere Game Loop. Die innere while(PeekMessage) verarbeitet alle anstehenden Windows-Messages auf einmal. Das ist wichtig, weil Windows manchmal mehrere Messages schnell hintereinander sendet (z. B. bei Mausbewegungen). Wenn wir nur eine pro Frame verarbeiten würden, würden sich Messages stapeln und das Spiel würde träge wirken.

Schritt 2: Window Procedure verstehen

Die MainWindowCallback ist die einzige Stelle, an der Windows mit dir spricht. Du musst hier mindestens drei Messages behandeln:

  • WM_CLOSE: Der Benutzer hat das X geklickt. Hier solltest du das Spiel beenden, aber nicht sofort das Fenster zerstören.
  • WM_DESTROY: Das Fenster wird gerade zerstört. Hier rufst du PostQuitMessage auf, was später eine WM_QUIT Message erzeugt.
  • WM_SIZE: Das Fenster wurde in der Größe verändert. Hier musst du später deinen Backbuffer anpassen.

Wichtig: Für jede Message, die du nicht selbst behandelst, musst du DefWindowProc aufrufen. Das ist die Standard-Implementierung von Windows. Ohne sie funktionieren grundlegende Dinge wie das Verschieben oder Minimieren des Fensters nicht mehr.

Tag 003–005: Der Backbuffer — warum wir nicht direkt auf den Bildschirm zeichnen

In Tag 003 stellt Casey eine fundamentale Frage: Wie bekommen wir Pixel in das Fenster? Die naive Antwort wäre: Windows gibt uns irgendeine Zeichenfunktion, und wir malen damit. Aber das ist viel zu langsam für ein Spiel. Stattdessen implementieren wir einen Backbuffer.

Was ist ein Backbuffer und warum brauchen wir ihn?

Stell dir vor, du zeichnest direkt auf den sichtbaren Bildschirm. Während du noch am Zeichnen bist, aktualisiert der Monitor das Bild. Das Ergebnis: Der Spieler sieht ein halb fertiges Bild — das nennt man Tearing oder Flackern. Ein Backbuffer löst das, indem er ein unsichtbares "Arbeitsblatt" bereitstellt. Du zeichnest das komplette Bild auf das Arbeitsblatt, und wenn es fertig ist, kopierst du es in einem einzigen, atomaren Schritt auf den Bildschirm.

Casey implementiert den Backbuffer als Device-Independent Bitmap (DIB). Das ist nichts anderes als ein großes Array von Bytes im Speicher, das Windows als Bitmap interpretieren kann. Jedes Pixel belegt 4 Bytes: Blau, Grün, Rot und ein Padding-Byte (das später als Alpha-Kanal genutzt wird).

🔷 Warum BGR und nicht RGB?

Windows verwendet aus historischen Gründen die Byte-Reihenfolge Blue-Green-Red (BGR) im Speicher. Wenn du Pixel-Daten Byte für Byte schreibst und dabei RGB-Reihenfolge annimmst, landest du bei vertauschten Rot- und Blau-Kanälen. Bei 32-Bit-Pixeln als uint32 ist das Layout 0xAARRGGBB — hier entspricht 0x00FF0000 tatsächlich Rot, weil Little-Endian und BGR sich gegenseitig aufheben. Der Fallstrick lauert vor allem beim direkten Byte-Zugriff über Pointer.

Schritt 3: Den Backbuffer allokieren

Wir brauchen einen Speicherbereich, der groß genug ist für alle Pixel unseres Fensters. Casey wählt hier bewusst VirtualAlloc statt malloc. Warum? Weil VirtualAlloc direkt mit dem Betriebssystem spricht. Es gibt keinen C-Runtime-Overhead, keine versteckten Buchhaltungsstrukturen. Du bekommst exakt den Speicher, den du angefordert hast, und er ist auf Null initialisiert (wenn du MEM_COMMIT verwendest).

Backbuffer-Allokation
struct win32_offscreen_buffer
{
    BITMAPINFO Info;
    void *Memory;
    int Width;
    int Height;
    int Pitch;
    int BytesPerPixel;
};

global_variable win32_offscreen_buffer GlobalBackbuffer;

internal void
Win32ResizeDIBSection(win32_offscreen_buffer *Buffer, int Width, int Height)
{
    // Wenn wir bereits einen Buffer haben, geben wir ihn frei
    if(Buffer->Memory)
    {
        VirtualFree(Buffer->Memory, 0, MEM_RELEASE);
    }
    
    Buffer->Width = Width;
    Buffer->Height = Height;
    Buffer->BytesPerPixel = 4;
    
    // Pitch = wie viele Bytes eine Zeile belegt
    // Hier einfach Width * 4, später könnte Alignment dazukommen
    Buffer->Pitch = Width * Buffer->BytesPerPixel;
    
    // BITMAPINFO-Header für Windows beschreiben
    Buffer->Info.bmiHeader.biSize = sizeof(Buffer->Info.bmiHeader);
    Buffer->Info.bmiHeader.biWidth = Buffer->Width;
    Buffer->Info.bmiHeader.biHeight = -Buffer->Height; // Negativ = top-down
    Buffer->Info.bmiHeader.biPlanes = 1;
    Buffer->Info.bmiHeader.biBitCount = 32;
    Buffer->Info.bmiHeader.biCompression = BI_RGB;
    
    // Speicher allokieren
    int BitmapMemorySize = (Buffer->Width * Buffer->Height) * Buffer->BytesPerPixel;
    Buffer->Memory = VirtualAlloc(0, BitmapMemorySize, MEM_RESERVE|MEM_COMMIT, PAGE_READWRITE);
}

Wichtige Details im Code

biHeight = -Buffer->Height: Das negative Vorzeichen ist entscheidend. In der Bitmap-Welt ist die Standard-Reihenfolge bottom-up: Die erste Zeile im Speicher ist die unterste Zeile des Bildes. Das ist mathematisch unintuitiv (die Y-Koordinate wächst nach oben). Mit einem negativen biHeight teilen wir Windows mit, dass unser Bild top-down ist: Die erste Zeile im Speicher ist die oberste Zeile. Das passt viel besser zu unserem mentalen Modell, bei dem Y nach unten wächst.

Pitch: Der Pitch beschreibt, wie viele Bytes eine komplette Zeile im Speicher belegt — einschließlich möglichen Paddings. Manche Grafik-APIs verlangen, dass jede Zeile auf eine bestimmte Byte-Grenze ausgerichtet ist (z. B. 4 Bytes oder 16 Bytes). In unserem einfachen Fall ist der Pitch einfach Width * 4, aber professionelle Renderer müssen hier oft Padding einfügen.

Schritt 4: Einen Pixel setzen

Um einen Pixel an Position (x, y) zu setzen, musst du seine Adresse im Speicher berechnen. Das ist die grundlegendste Operation jedes Software-Renderers. Verstehe diese Formel — du wirst sie Tausende Male verwenden.

Pixel-Zugriff
uint32 *Pixel = (uint32 *)((uint8 *)Buffer.Memory 
    + x * Buffer.BytesPerPixel 
    + y * Buffer.Pitch);
    
*Pixel = 0xFF00FF; // Magenta in BGR (Blau + Rot)

Die Berechnung funktioniert so: Wir starten am Anfang des Speicherblocks (Buffer.Memory). Dann springen wir y * Pitch Bytes nach unten, um zur richtigen Zeile zu gelangen. Dann springen wir x * BytesPerPixel Bytes nach rechts, um zur richtigen Spalte zu gelangen. Das Ergebnis ist ein Pointer auf den Anfang des 4-Byte-Pixels.

Den Buffer auf den Bildschirm bringen

Das Zeichnen in den Speicher allein reicht nicht — wir müssen Windows sagen, dass es diesen Speicher in das Fenster kopieren soll. Dafür verwenden wir StretchDIBits oder später BitBlt. Casey wählt StretchDIBits, weil es uns erlaubt, den Buffer auf die Client-Area des Fensters zu skalieren — auch wenn das Fenster eine andere Größe hat als unser Buffer.

Buffer anzeigen
internal void
Win32DisplayBufferInWindow(HDC DeviceContext, int WindowWidth, int WindowHeight,
                            win32_offscreen_buffer *Buffer)
{
    StretchDIBits(DeviceContext,
                  0, 0, WindowWidth, WindowHeight,    // Ziel-Rechteck
                  0, 0, Buffer->Width, Buffer->Height, // Quell-Rechteck
                  Buffer->Memory,
                  &Buffer->Info,
                  DIB_RGB_COLORS,
                  SRCCOPY);
}

In Tag 004 wird dieser Buffer animiert: Wir zeichnen ein sich bewegendes Rechteck, indem wir in jedem Frame die X-Position erhöhen und den Buffer komplett neu füllen. Das ist der erste "Game Loop" im klassischen Sinne: Input → Update → Render. Beachte, dass wir den gesamten Buffer in jedem Frame neu zeichnen müssen. Es gibt kein "nur das bewegte Objekt aktualisieren" — das wäre viel zu komplex und fehleranfällig. Stattdessen rendern wir alles from scratch.

Tag 006–010: Input, Sound & Timing — die Sinne des Spiels

Keyboard & Gamepad (Tag 006)

Windows-Keyboard-Input kommt über WM_KEYDOWN und WM_KEYUP. Casey entscheidet sich aber früh dagegen, diese Messages direkt für Spielsteuerung zu verwenden. Warum? Weil Windows ein komisches Verhalten hat: Wenn du eine Taste gedrückt hältst, sendet Windows zunächst ein WM_KEYDOWN, wartet eine halbe Sekunde, und sendet dann wiederholte WM_KEYDOWN-Messages. Das ist perfekt für Textverarbeitung, aber tödlich für Spiele. Wenn ein Spieler die Sprung-Taste drückt und hält, willst du nicht, dass der Charakter erst eine halbe Sekunde später reagiert.

Schritt 5: Einen Key-State-Array bauen

Die Lösung ist einfach und elegant: Wir verwenden die Windows-Messages nur, um einen internen Zustand zu aktualisieren. Jede Frame lesen wir diesen Zustand aus — nicht die Messages.

Input-Struktur
struct game_button_state
{
    int HalfTransitionCount;
    bool32 EndedDown;
};

struct game_controller_input
{
    bool32 IsConnected;
    bool32 IsAnalog;
    
    game_button_state MoveUp;
    game_button_state MoveDown;
    game_button_state MoveLeft;
    game_button_state MoveRight;
    game_button_state ActionUp;
    game_button_state ActionDown;
    // ... weitere Buttons
    
    real32 StickAverageX;
    real32 StickAverageY;
};

struct game_input
{
    game_controller_input Controllers[5];
};

Beachte das Feld HalfTransitionCount: Es zählt, wie oft der Button seinen Zustand in diesem Frame gewechselt hat. Das klingt übertrieben, aber es ist wichtig für schnelle Eingaben. Wenn ein Spieler sehr schnell drückt und loslässt, kann es passieren, dass beide Events (KEYDOWN und KEYUP) zwischen zwei Frames stattfinden. Ohne HalfTransitionCount würdest du den Tastendruck komplett verpassen!

XInput und dynamisches Laden (Tag 006)

Für Gamepad-Input verwenden wir XInput, die Xbox-Controller-API von Microsoft. Aber hier gibt es einen Haken: Ältere Windows-Versionen (vor Windows 8) haben XInput nicht standardmäßig installiert. Wenn wir direkt gegen die XInput-Library linken würden, würde unser Spiel auf diesen Systemen gar nicht starten.

Caseys Lösung: Dynamisches Laden zur Laufzeit. Wir laden die XInput-DLL mit LoadLibrary und holen uns die Funktionspointer mit GetProcAddress. Wenn XInput nicht vorhanden ist, setzen wir einfach alle Controller-Inputs auf Null. Das Spiel läuft weiter — nur ohne Gamepad-Unterstützung.

DirectSound (Tag 007–009)

Audio ist in Spielen kritisch und wird oft unterschätzt. Casey initialisiert DirectSound und erklärt dabei die grundlegende Architektur von Windows-Audio.

🔊 Primary Buffer vs. Secondary Buffer

DirectSound hat zwei Arten von Buffern: Der Primary Buffer ist direkt mit der Soundkarte verbunden. Es gibt ihn nur einmal pro Anwendung. Der Secondary Buffer enthält deine eigentlichen Audiodaten (Musik, Effekte). DirectSound mischt alle Secondary Buffers zusammen und spielt das Ergebnis über den Primary Buffer ab. In unserem Fall verwenden wir aber gar nicht den Mixer von DirectSound, sondern schreiben direkt in einen großen Secondary Buffer, den wir selbst mischen.

Casey implementiert einen Ringbuffer für Audio. Das ist ein kontinuierlicher Speicherbereich, der logisch als Kreis betrachtet wird. Wir schreiben Audio-Daten hinein, und die Soundkarte liest sie an einer anderen Position aus. Wenn der Schreib- oder Lese-Pointer das Ende des Speichers erreicht, springt er wieder zum Anfang.

Audio-Ringbuffer
struct win32_sound_output
{
    int SamplesPerSecond;
    int ToneHz;
    int ToneVolume;
    uint32 RunningSampleIndex;
    int WavePeriod;
    int BytesPerSample;
    int SecondaryBufferSize;
    real32 tSine;
    int LatencySampleCount;
};

In Tag 008 wird eine Square Wave generiert. Das ist die einfachste mögliche Schwingung: Der Wert springt zwischen einem positiven und einem negativen Maximum. Sie klingt wie ein klassisches 8-Bit-Computerspiel. In Tag 009 wird das zu einer Sine Wave mit variabler Tonhöhe erweitert. Die mathematische Formel ist:

sample = sin(2π · frequency · t)

Casey berechnet den Sinus inkrementell, um teure sin()-Aufrufe pro Sample zu vermeiden. Er speichert den aktuellen Phasenwinkel (tSine) und erhöht ihn in jedem Schritt um 2π / WavePeriod.

Timing (Tag 010)

Um die Geschwindigkeit des Spiels unabhängig von der Framerate zu machen, brauchen wir präzises Timing. Casey führt QueryPerformanceCounter ein — die hochpräzise Hardware-Zeitquelle von Windows. Dieser Counter nutzt einen festen Hardware-Timer (je nach System HPET, TSC oder ACPI PM Timer) und hat eine Auflösung im Mikro- bis Nanosekunden-Bereich. Die genaue Frequenz ermittelt man mit QueryPerformanceFrequency.

Schritt 6: Delta Time berechnen

In deiner Game Loop solltest du niemals davon ausgehen, dass ein Frame exakt 16.67ms dauert (60 FPS). Die tatsächliche Zeit kann stark schwanken. Berechne stattdessen die verstrichene Zeit seit dem letzten Frame und verwende diese als Basis für alle Bewegungen und Animationen.

Game Loop mit Delta Time
LARGE_INTEGER PerfCountFrequencyResult;
QueryPerformanceFrequency(&PerfCountFrequencyResult);
int64 PerfCountFrequency = PerfCountFrequencyResult.QuadPart;

LARGE_INTEGER LastCounter = GetWallClock();
while(Running)
{
    // ... Input verarbeiten ...
    
    LARGE_INTEGER WorkCounter = GetWallClock();
    real32 DeltaTime = (real32)(WorkCounter.QuadPart - LastCounter.QuadPart) 
                    / (real32)PerfCountFrequency;
    
    // DeltaTime ist jetzt in Sekunden, z.B. 0.0167 für 60 FPS
    GameUpdateAndRender(DeltaTime, ...);
    
    LastCounter = WorkCounter;
}

Casey erwähnt auch RDTSC (Read Time-Stamp Counter), einen CPU-Befehl, der noch schneller ist als QueryPerformanceCounter. Aber er warnt davor: Moderne CPUs ändern ihre Taktfrequenz dynamisch (Turbo Boost, Energiesparmodus), und auf Multi-Core-Systemen können die Time-Stamp-Counter der verschiedenen Kerne auseinanderlaufen. QueryPerformanceCounter ist langsamer, aber zuverlässig.

Tag 011–025: Plattformunabhängigkeit & Hot Reloading

Ab Tag 011 beginnt der wichtigste architektonische Schritt der gesamten Serie: Die Trennung zwischen Plattform-Code (Win32-spezifisch) und Spiel-Code (plattformunabhängig). Diese Trennung ist nicht nur akademisch — sie ist das Fundament für das berühmte Hot Code Reloading.

Die API-Grenze definieren

Casey definiert eine klare API zwischen Plattform und Spiel. Das Spiel bekommt von der Plattform folgende Dinge:

  • Memory: Ein großer Speicherblock, den das Spiel selbst verwaltet
  • Input: Tastatur- und Controller-States in einer normalisierten Struktur
  • Sound: Ein Buffer, den das Spiel mit Audio-Samples füllt
  • File I/O: Lese-/Schreiboperationen für Debug-Zwecke

Das Spiel gibt zurück:

  • Bitmap-Daten: Ein gefüllter Backbuffer, den die Plattform auf den Bildschirm zeichnet
  • Audio-Daten: Ein gefüllter Sound-Buffer

⚠️ Kein malloc im Spiel-Code!

Eine der wichtigsten Design-Entscheidungen: Das Spiel darf niemals selbst malloc, new oder VirtualAlloc aufrufen. Die Plattform allokiert einmalig einen großen Speicherblock (z. B. 1 GB für permanenten Speicher und einige MB für transienten Speicher) und übergibt ihn an das Spiel. Das Spiel verwaltet diesen Speicher intern mit Memory Arenas oder einfachen Offset-Allocatoren.

Warum? Weil dynamische Allokation während des Gameplays unvorhersehbar langsam ist. malloc kann intern Locks verwenden, Speicher vom OS anfordern, oder Garbage Collection auslösen. In einem Spiel, das 60 Mal pro Sekunde ein festes Zeitbudget hat, ist das inakzeptabel.

Memory Arenas erklärt

Eine Memory Arena ist der einfachste mögliche Allocator: Du hast einen großen Speicherblock und einen Offset-Zeiger. Wenn du Speicher brauchst, erhöhst du den Zeiger um die gewünschte Größe und gibst die alte Position zurück. Das ist O(1) und extrem cache-freundlich.

Einfache Memory Arena
struct memory_arena
{
    memory_index Size;
    uint8 *Base;
    memory_index Used;
};

void *PushSize_(memory_arena *Arena, memory_index Size)
{
    Assert((Arena->Used + Size) <= Arena->Size);
    void *Result = Arena->Base + Arena->Used;
    Arena->Used += Size;
    
    return(Result);
}

#define PushStruct(Arena, type) (type *)PushSize_(Arena, sizeof(type))
#define PushArray(Arena, Count, type) (type *)PushSize_(Arena, (Count)*sizeof(type))

Beachte: Eine Arena kann nur allokieren, nicht freigeben (außer du setzt Used zurück). Das klingt wie eine Einschränkung, ist aber für Spiele oft perfekt: Du allokierst alles, was du für einen Frame brauchst, und am Ende des Frames setzt du Used auf Null zurück. Das ist der schnellste "Garbage Collector", den es gibt.

Hot Code Reloading (Tag 021–023)

Jetzt kommt einer der revolutionärsten Aspekte von Handmade Hero: Hot Code Reloading. Die Plattform lädt das Spiel nicht als statisch gelinkten Code, sondern als dynamische Bibliothek (handmade.dll). Wenn sich die DLL ändert — weil wir neu kompiliert haben — entlädt die Plattform die alte Version und lädt die neue. Das Spiel läuft weiter.

Schritt 7: Hot Reloading implementieren

Das ist der Punkt, an dem Handmade Hero sich von fast allen anderen Tutorials unterscheidet. Du wirst in der Lage sein, den Spielcode zu ändern, neu zu kompilieren, und das Ergebnis sofort im laufenden Spiel zu sehen — ohne Neustart, ohne Level-Neuladen.

Live Code Reloading
struct win32_game_code
{
    HMODULE GameCodeDLL;
    FILETIME DLLLastWriteTime;
    
    game_update_and_render *UpdateAndRender;
    game_get_sound_samples *GetSoundSamples;
    
    bool32 IsValid;
};

internal win32_game_code
Win32LoadGameCode(char *SourceDLLName, char *TempDLLName)
{
    win32_game_code Result = {};
    
    // Windows sperrt geladene DLLs, also kopieren wir sie zuerst
    CopyFile(SourceDLLName, TempDLLName, FALSE);
    
    Result.GameCodeDLL = LoadLibraryA(TempDLLName);
    if(Result.GameCodeDLL)
    {
        Result.UpdateAndRender = (game_update_and_render *)
            GetProcAddress(Result.GameCodeDLL, "GameUpdateAndRender");
        Result.GetSoundSamples = (game_get_sound_samples *)
            GetProcAddress(Result.GameCodeDLL, "GameGetSoundSamples");
        
        Result.IsValid = (Result.UpdateAndRender && Result.GetSoundSamples);
    }
    
    if(!Result.IsValid)
    {
        Result.UpdateAndRender = 0;
        Result.GetSoundSamples = 0;
    }
    
    return(Result);
}

// In der Game Loop:
FILETIME NewDLLWriteTime = GetLastWriteTime(SourceDLLName);
if(CompareFileTime(&NewDLLWriteTime, &Game.DLLLastWriteTime) != 0)
{
    Win32UnloadGameCode(&Game);
    Game = Win32LoadGameCode(SourceDLLName, TempDLLName);
    Game.DLLLastWriteTime = NewDLLWriteTime;
}

Warum die Kopie? Windows sperrt geladene DLL-Dateien. Wenn der Compiler versucht, handmade.dll zu überschreiben, während sie geladen ist, schlägt das fehl. Die Lösung ist einfach: Wir kopieren die DLL zu handmade_temp.dll und laden diese Kopie. Das Original bleibt frei zum Überschreiben.

Was passiert mit dem Spielzustand? Das ist der geniale Teil: Der Spielzustand lebt im Speicher der Plattform, nicht in der DLL. Wenn wir die DLL neu laden, bleibt der Speicherblock mit allen Entities, der Spielerposition, der Welt usw. vollständig erhalten. Das neue Code läuft einfach auf den alten Daten weiter.

In der Software-Industrie gibt es einen gefährlichen Trend: Entwickler lernen ein Muster (z. B. Transient State vs. Permanent State

Eine der ersten architektonischen Entscheidungen ist die Trennung zwischen zwei Arten von Zustand:

  • Permanent State: Muss über Frames hinweg persistieren. Wenn das Spiel pausiert wird und später fortgesetzt wird, muss dieser Zustand noch da sein. Beispiele: Spieler-Position, Gesundheit, Inventar, Welt-Layout.
  • Transient State: Kann jeder Frame neu berechnet werden. Wenn wir den Frame neu starten würden, wäre dieser Zustand identisch. Beispiele: Render-Listen, temporäre Kollisions-Listen, Partikel-Systeme, Sound-Mix-Buffer.

Diese Trennung ist mächtig, weil sie uns erlaubt, den transienten Speicher extrem aggressiv zu verwalten. Wir können eine Memory Arena für transienten Speicher verwenden und sie am Ende jedes Frames komplett zurücksetzen. Das ist schneller und einfacher als jedes Garbage-Collection-System.

Tag 028–035: Tilemaps & Koordinatensysteme

Ein Tilemap ist ein 2D-Gitter, bei dem jede Zelle einen bestimmten Typ hat (z. B. Wand, Boden, Tür). Für viele 2D-Spiele ist das die effizienteste und flexibelste Art, Welten darzustellen. Casey beginnt mit einer sehr einfachen Darstellung: Ein 2D-Array von uint32-Werten. Aber schnell wird klar, dass feste Arrays zu unflexibel sind.

Das Problem mit festen Arrays

Wenn du ein festes Array wie uint32 Tiles[256][256] verwendest, bist du auf eine Welt von 256×256 Tiles beschränkt. Das mag für ein kleines Level ausreichen, aber für ein Open-World-Spiel oder sogar nur ein Spiel mit mehreren verbundenen Levels ist das unbrauchbar. Außerdem verschwendest du Speicher: Wenn dein Level nur 50×50 Tiles groß ist, bleiben die restlichen 65.000 Tiles ungenutzt.

Schritt 8: Von festen Arrays zu virtuellen Tilemaps

In Tag 033 führt Casey das Konzept der virtuellen Tilemap ein. Die Idee ist elegant: Die Welt ist potenziell unendlich groß. Wir speichern nicht die gesamte Welt, sondern nur die Chunks — kleine Kachelbereiche — die gerade relevant sind. Ein Chunk ist z. B. 16×16 oder 32×32 Tiles groß.

Was du tun sollst: Statt ein großes 2D-Array zu verwenden, teile deine Welt in Chunks auf. Berechne für jede Welt-Position, zu welchem Chunk sie gehört, und speichere nur die Chunks, die tatsächlich existieren.

Tilemap-Koordinaten
struct tile_map
{
    uint32 *Tiles;      // Speicher für alle geladenen Chunks
    uint32 ChunkShift;  // log2(ChunkDim) für schnelle Division
    uint32 ChunkMask;   // ChunkDim - 1 für Modulo
    uint32 ChunkDim;    // z.B. 16
    
    uint32 CountX;
    uint32 CountY;
    
    real32 TileSideInMeters;
};

// Welt-Koordinaten → Chunk + lokale Tile-Koordinaten
inline tile_chunk_position
GetChunkPositionFor(tile_map *TileMap, int32 AbsTileX, int32 AbsTileY)
{
    tile_chunk_position Result;
    
    Result.TileChunkX = AbsTileX >> TileMap->ChunkShift;
    Result.TileChunkY = AbsTileY >> TileMap->ChunkShift;
    Result.RelTileX = AbsTileX & TileMap->ChunkMask;
    Result.RelTileY = AbsTileY & TileMap->ChunkMask;
    
    return(Result);
}

Beachte den Trick mit ChunkShift und ChunkMask: Wenn die Chunk-Größe eine Potenz von 2 ist (z. B. 16 = 2⁴), können wir Division und Modulo durch Bit-Shifts und Bit-Masks ersetzen. Das ist um Größenordnungen schneller als eine echte Integer-Division, die auf den meisten CPUs mehrere Taktzyklen benötigt.

Warum Potenzen von 2? Weil Computer im Binärsystem arbeiten. Ein Shift um 4 Bits nach rechts ist exakt äquivalent zu einer Division durch 16. Eine AND-Maske mit 0xF (15) ist exakt äquivalent zu mod 16. Diese Operationen werden in einem einzigen CPU-Takt ausgeführt.

Sparse Storage (Tag 035)

In Tag 035 wird das Speichersystem weiter optimiert. Statt ein großes Array für alle Chunks zu haben, verwenden wir ein Hash-basiertes Sparse Array. Nur Chunks, die tatsächlich existieren, belegen Speicher. Das ermöglicht riesige, nahezu unendliche Welten ohne massiven Speicherverbrauch.

Die Idee ist einfach: Wir haben ein Array von Chunk-Pointern. Jeder Pointer ist entweder NULL (der Chunk existiert nicht) oder zeigt auf einen tatsächlich allokierten Chunk. Wenn ein Spieler in einen neuen Bereich läuft, allokieren wir den Chunk dynamisch (aus unserer Memory Arena) und füllen ihn mit Daten.

💡 Der Vorteil von Sparse Arrays

Stell dir eine Welt vor, die theoretisch 1.000.000×1.000.000 Tiles groß ist. Mit einem festen Array würdest du 4 Billionen Tiles speichern müssen — das sind Terabytes an Speicher. Mit einem Sparse Array speicherst du nur die Chunks, in denen sich der Spieler gerade aufhält und die in der Nähe sind. Vielleicht 100 Chunks à 256 Tiles. Das sind gerade einmal 25.600 Tiles.

Tag 029–032: Kollisionserkennung — die Mathematik der Bewegung

Kollision ist das Herzstück jedes Spiels mit Bewegung. Ohne Kollision würde der Spieler durch Wände laufen, in Abgründe fallen oder in Gegner hineinphasing. Casey beginnt mit der einfachsten Form: AABB-Kollision (Axis-Aligned Bounding Box).

AABB-Kollision

Eine AABB ist ein Rechteck, dessen Seiten parallel zu den Koordinatenachsen verlaufen. Das macht den Kollisionstest extrem einfach: Zwei Rechtecke kollidieren, wenn sie sich auf der X-Achse und auf der Y-Achse überlappen.

AABB-Überlappungstest
bool32 RectsIntersect(rectangle2 A, rectangle2 B)
{
    bool32 Result = !((B.Max.x <= A.Min.x) ||
                     (B.Min.x >= A.Max.x) ||
                     (B.Max.y <= A.Min.y) ||
                     (B.Min.y >= A.Max.y));
    return(Result);
}

Dieser Test ist bemerkenswert effizient: Er verwendet nur vier Vergleiche und eine logische Negation. Aber er hat ein großes Problem: Er sagt uns nur ob eine Kollision stattfindet, nicht wie weit die Objekte ineinander eingedrungen sind. Für eine saubere Kollisionsauflösung brauchen wir mehr Informationen.

Kanonische Positionen

Casey führt hier das Konzept der kanonischen Koordinaten ein: Jede Position wird in einen Tile-Index und einen Offset innerhalb dieses Tiles zerlegt. Das ist nicht nur eine nette Datenstruktur — es ist eine fundamentale Design-Entscheidung, die die Präzision und Skalierbarkeit der Engine bestimmt.

Kanonische Position
struct tile_map_position
{
    uint32 AbsTileX;
    uint32 AbsTileY;
    uint32 AbsTileZ;
    
    real32 OffsetX; // Offset innerhalb des Tiles [0, TileSide)
    real32 OffsetY;
};

Diese Darstellung ist mächtig, weil sie unendliche Welten unterstützt, ohne Floating-Point-Präzisionsprobleme zu bekommen. Die grobe Position ist in Tile-Indizes gespeichert (Integer, also absolut präzise), und nur die feine Position innerhalb eines Tiles ist ein Float. Selbst wenn der Spieler Millionen von Tiles weit läuft, verliert er nie an Präzision, weil die Tile-Indizes exakt sind.

Warum ist das wichtig? In einem typischen 3D-Spiel verwendest du reine Floating-Point-Koordinaten. Wenn der Spieler sehr weit vom Ursprung entfernt ist (z. B. in einem Open-World-Spiel), reichen die verfügbaren Bits der Float nicht mehr aus, um kleine Bewegungen darzustellen. Das führt zu Ruckeln und Kollisionsfehlern — ein berühmtes Problem, das in vielen kommerziellen Spielen auftritt. Die kanonische Position vermeidet das komplett.

Tag 036–055: Entities & Welt-Speicherung

Ab Tag 036 kommen Entities ins Spiel — alles, was sich bewegt oder interagiert: der Spieler, Gegner, Projektile, Schalter, Türen. Casey implementiert ein sehr einfaches Entity-Array. Das ist bewusst so gewählt.

Warum kein ECS-Framework?

In der modernen Game-Development-Szene ist ECS (Entity-Component-System) extrem populär. Unity hat es adoptiert, Unreal hat ähnliche Konzepte, und fast jedes Tutorial im Internet predigt ECS als die "eine wahre Architektur". Casey lehnt es für dieses Projekt ab — und zwar aus sehr guten Gründen.

ECS ist mächtig, wenn du Tausende von Entities mit komplexen, dynamischen Beziehungen hast. Aber es kommt mit massivem Overhead: Du brauchst Component-Manager, System-Registrierung, Entity-Handles, Archetypes, Chunk-Allokation, Query-Caches... Für ein kleines bis mittleres 2D-Spiel ist das alles völlig überdimensioniert. Ein flaches Array von Structs ist nicht nur einfacher, sondern auch cache-freundlicher, weil alle Daten eines Entities dicht beieinander im Speicher liegen.

Entity-Struktur
struct entity
{
    uint32 Exists;
    tile_map_position P;
    tile_map_position dP; // Geschwindigkeit
    
    uint32 FacingDirection;
    real32 Width;
    real32 Height;
    
    uint32 Collision;
    
    // ... weitere Eigenschaften
};

#define MAX_ENTITY_COUNT 4096
struct game_state
{
    entity Entities[MAX_ENTITY_COUNT];
    uint32 EntityCount;
    // ...
};

Beachte das Feld Exists: Es ist ein einfacher Boolean, der sagt, ob dieser Slot im Array belegt ist. Wenn eine Entity "gelöscht" wird, setzen wir einfach Exists = false. Der Slot kann später von einer neuen Entity wiederverwendet werden. Das ist viel schneller als ein dynamisches Array zu verkleinern oder Elemente zu verschieben.

Hash-basierte Welt-Speicherung (Tag 055)

Wenn Entities sich frei in der Welt bewegen, können wir sie nicht mehr in einem einfachen 2D-Array nach Position indizieren. Wenn wir wissen wollen, welche Entities sich in der Nähe des Spielers befinden, können wir nicht das gesamte Array durchsuchen — das wäre O(n) und würde bei Tausenden von Entities extrem langsam werden.

Casey implementiert daher einen Spatial Hash: Die Welt wird in Chunks unterteilt, und jeder Chunk hat eine Liste der Entities, die sich gerade in ihm befinden. Wenn sich eine Entity bewegt, wird sie aus dem alten Chunk entfernt und in den neuen eingefügt.

💡 Warum kein Quadtree?

Casey diskutiert verschiedene räumliche Datenstrukturen und entscheidet sich gegen komplexe Bäume. Für ein Spiel mit relativ gleichmäßig verteilten Entities und einer festen Chunk-Größe ist ein Hash-basiertes Grid schneller, einfacher und cache-freundlicher. Bäume lohnen sich erst bei extrem ungleichmäßiger Verteilung (z. B. wenn 90% der Entities in einem kleinen Bereich konzentriert sind).

Ein Quadtree hat außerdem einen konstanten Update-Overhead: Wenn sich eine Entity bewegt, musst du möglicherweise den Baum rebalancieren oder Knoten teilen/verschmelzen. Ein Hash-Grid hat diesen Overhead nicht: Du berechnest den neuen Chunk-Index und fügst die Entity in eine Linked List ein. Das ist O(1).

Bis Ende Phase 2 haben wir ein funktionierendes Grundgerüst: Ein Spieler, der sich in einer Tilemap bewegt, mit Wand-Kollision, einem einfachen aber effektiven Entity-System und einer skalierbaren, speicher-effizienten Welt-Speicherung. Die Engine ist bereit für die nächste Stufe: Professionelles Rendering.

Tag 036–040: Bitmap-Loading & Rendering

Casey entscheidet sich bewusst für das BMP-Format als erstes Bildformat. Warum nicht PNG? Warum nicht JPEG? Die Antwort ist wieder: Kontrolle und Verständnis.

Warum BMP?

BMP ist unkomprimiert und hat eine extrem einfache Header-Struktur. Wir können es komplett selbst parsen, ohne externe Bibliotheken wie libpng oder stb_image zu verwenden. PNG ist komprimiert (mit DEFLATE), was bedeutet, dass wir einen Dekompressions-Algorithmus implementieren müssten, bevor wir auch nur ein einziges Pixel auf den Bildschirm bringen können. Casey wird später (Tag 453–458) tatsächlich einen PNG-Parser bauen — aber zu diesem Zeitpunkt würde es den Lernprozess nur verlangsamen.

Schritt 9: Einen BMP-Loader von Grund auf schreiben

Öffne eine Bilddatei im Hex-Editor (oder verwende die Datei, die Casey bereitstellt). Die ersten 14 Bytes sind der BMP-File-Header, gefolgt vom DIB-Header (mindestens 40 Bytes). Du musst diese Strukturen exakt nachbilden.

Wichtig: Verwende #pragma pack(push, 1), um sicherzustellen, dass der Compiler keine Padding-Bytes zwischen den Struct-Members einfügt. Ohne das würde dein Header nicht exakt der Spezifikation entsprechen.

BMP-Header (vereinfacht)
#pragma pack(push, 1)
struct bitmap_header
{
    uint16 FileType;     // 'BM'
    uint32 FileSize;
    uint16 Reserved1;
    uint16 Reserved2;
    uint32 BitmapOffset; // Wo beginnen die Pixel-Daten?
    uint32 Size;         // Größe des Info-Headers (40)
    int32 Width;
    int32 Height;
    uint16 Planes;
    uint16 BitsPerPixel; // 24 oder 32
    uint32 Compression;
    uint32 SizeOfBitmap;
    int32 HorzResolution;
    int32 VertResolution;
    uint32 ColorsUsed;
    uint32 ColorsImportant;
};
#pragma pack(pop)

struct loaded_bitmap
{
    int32 Width;
    int32 Height;
    uint32 *Pixels;
};

#pragma pack(push, 1) ist hier entscheidend: Der C-Compiler fügt normalerweise Padding-Bytes in Structs ein, um die Member auf natürliche Alignment-Grenzen auszurichten (z. B. 4-Byte-Grenzen für int32). Das ist für die Performance gut, aber für das Parsen von Binary-Dateien tödlich. Der BMP-Header hat exakt definierte Offsets. Wenn der Compiler Padding einfügt, liest du die falschen Bytes.

Bitmap Blitting

Das Kopieren einer Bitmap in den Backbuffer nennt man Blitting (Block Image Transfer). Die erste Implementierung ist ein einfaches Pixel-für-Pixel-Kopieren. Für jede Zeile der Quell-Bitmap kopieren wir die Pixel in die entsprechende Zeile des Ziel-Backbuffers.

Einfaches Bitmap-Blitting
internal void
DrawBitmap(game_offscreen_buffer *Buffer, loaded_bitmap *Bitmap, 
           real32 RealX, real32 RealY)
{
    int32 X = RoundReal32ToInt32(RealX);
    int32 Y = RoundReal32ToInt32(RealY);
    
    int32 MinX = X;
    int32 MinY = Y;
    int32 MaxX = X + Bitmap->Width;
    int32 MaxY = Y + Bitmap->Height;
    
    // Clipping: Stelle sicher, dass wir nicht außerhalb des Buffers zeichnen
    if(MinX < 0) { MinX = 0; }
    if(MinY < 0) { MinY = 0; }
    if(MaxX > Buffer->Width) { MaxX = Buffer->Width; }
    if(MaxY > Buffer->Height) { MaxY = Buffer->Height; }
    
    uint32 *SourceRow = Bitmap->Pixels + (Bitmap->Height - 1) * Bitmap->Width;
    uint8 *DestRow = ((uint8 *)Buffer->Memory +
                          MinX * Buffer->BytesPerPixel +
                          MinY * Buffer->Pitch);
    
    for(int Y = MinY; Y < MaxY; ++Y)
    {
        uint32 *Dest = (uint32 *)DestRow;
        uint32 *Source = SourceRow;
        for(int X = MinX; X < MaxX; ++X)
        {
            *Dest++ = *Source++;
        }
        
        DestRow += Buffer->Pitch;
        SourceRow -= Bitmap->Width; // BMP ist bottom-up!
    }
}

Beachte das SourceRow -= Bitmap->Width: BMP-Dateien speichern ihre Pixel in bottom-up-Reihenfolge. Die erste Zeile im Speicher ist die unterste Zeile des Bildes. Wenn wir das Bild korrekt herauszeichnen wollen, müssen wir die Quell-Zeilen rückwärts durchlaufen.

Tag 038–039: Alpha-Blending — die Mathematik der Transparenz

Einfaches Kopieren reicht nicht für professionelle Grafik. Wenn wir einen Charakter vor einem Hintergrund zeichnen wollen, müssen die Pixel des Charakters mit den Pixeln des Hintergrunds gemischt werden — basierend auf dem Alpha-Wert (Transparenz).

Was ist Alpha-Blending?

Jedes Pixel hat vier Kanäle: Rot, Grün, Blau und Alpha. Der Alpha-Kanal bestimmt, wie undurchsichtig das Pixel ist. Ein Alpha-Wert von 255 bedeutet vollständig undurchsichtig, 0 bedeutet vollständig transparent. Wenn wir ein Quell-Pixel (Source) über ein Ziel-Pixel (Destination) zeichnen, berechnen wir die resultierende Farbe als gewichtetes Mittel.

Lineares Alpha-Blending
real32 Inv255 = 1.0f / 255.0f;

// Quell-Pixel (das, was wir zeichnen wollen)
real32 SA = (real32)((Source >> 24) & 0xFF) * Inv255;
real32 SR = (real32)((Source >> 16) & 0xFF) * Inv255;
real32 SG = (real32)((Source >> 8) & 0xFF) * Inv255;
real32 SB = (real32)(Source & 0xFF) * Inv255;

// Ziel-Pixel (das, was bereits im Buffer ist)
real32 DA = (real32)((Dest >> 24) & 0xFF) * Inv255;
real32 DR = (real32)((Dest >> 16) & 0xFF) * Inv255;
real32 DG = (real32)((Dest >> 8) & 0xFF) * Inv255;
real32 DB = (real32)(Dest & 0xFF) * Inv255;

// Standard-Porter-Duff "Over" Operation
real32 R = (1.0f - SA) * DR + SR;
real32 G = (1.0f - SA) * DG + SG;
real32 B = (1.0f - SA) * DB + SB;
real32 A = (1.0f - SA) * DA + SA;

// Zurück in 8-Bit-Integer konvertieren
uint32 Result = ((uint32)(A * 255.0f) << 24) |
                ((uint32)(R * 255.0f) << 16) |
                ((uint32)(G * 255.0f) << 8)  |
                ((uint32)(B * 255.0f));

Die Formel (1.0f - SA) * DR + SR ist die klassische Over-Operation aus dem Porter-Duff-Compositing-Modell. Sie besagt: Das Ergebnis ist der alte Hintergrund, abgedunkelt um den Anteil des Vordergrunds, plus der Vordergrund selbst. Wenn der Vordergrund vollständig undurchsichtig ist (SA = 1.0), verschwindet der Hintergrund komplett. Wenn er vollständig transparent ist (SA = 0.0), bleibt nur der Hintergrund.

⚠️ Das sRGB-Problem

Das obige Blending findet im linearen Farbraum statt. Das ist mathematisch korrekt, aber die meisten Bilder (BMPs, PNGs, JPEGs) speichern ihre Farben in sRGB — einem nicht-linearen Farbraum, der für menschliche Wahrnehmung optimiert ist. Wenn du sRGB-Werte direkt mischt, entstehen falsche Farbübergänge und dunkle Ränder um halbtransparente Objekte.

Casey kommt in Tag 094 auf dieses Thema zurück und implementiert Gamma-korrektes Blending. Für den Moment ist das lineare Blending ausreichend, aber du solltest im Hinterkopf behalten, dass professionelle Renderer einen sRGB-zu-linear-Konversionsschritt durchführen, bevor sie blenden.

2D-Vektoren

In Tag 042 werden grundlegende Vektor-Operationen eingeführt. Vektoren sind das mathematische Werkzeug für Bewegung, Richtung, Kraft und Position. Casey implementiert sie als C++-Union, die sowohl über .x und .y als auch über ein Array .E[2] zugänglich ist.

2D-Vektor-Struktur
union v2
{

    {
        real32 x, y;
    };
    struct

    };

Casey verwendet hier bewusst C++-Operator-Overloading, obwohl der Rest des Codes C-ähnlich ist. Das ist eine der wenigen C++-Features, die er akzeptiert, weil sie den Mathe-Code lesbarer machen, ohne Runtime-Overhead zu erzeugen. A + B ist viel intuitiver als V2Add(A, B).

Das Skalarprodukt (Dot Product)

Das Dot Product ist eine der wichtigsten Operationen in der gesamten Computergrafik und Spielphysik. Es hat zwei fundamentale Interpretationen:

  1. Projektion: Dot(A, B) gibt die Länge der Projektion von A auf B an (skaliert mit der Länge von B).
  2. Winkel: Dot(A, B) = |A| · |B| · cos(θ). Wenn das Dot Product positiv ist, zeigen die Vektoren in ähnliche Richtungen. Wenn es negativ ist, zeigen sie in entgegengesetzte Richtungen. Wenn es Null ist, sind sie orthogonal.

In Spielen verwendet man das Dot Product ständig: Für Beleuchtungsberechnungen (wie viel Licht trifft auf eine Oberfläche?), für Kollisionsauflösung (in welche Richtung soll ich zurückschieben?), für KI (ist der Gegner vor mir oder hinter mir?).

Bewegungsgleichungen (Tag 043)

Die Physik des Spiels ist bewusst einfach gehalten. Casey implementiert die grundlegenden Bewegungsgleichungen der klassischen Mechanik:

  • Position += Velocity × dt
  • Velocity += Acceleration × dt

Das ist sogenannte Euler-Integration. Sie ist nicht die präziseste numerische Integration (für starke Kräfte oder große Zeitschritte kann sie instabil werden), aber für einfache Spiele mit kleinen dt ist sie völlig ausreichend. Casey erwähnt, dass für komplexere Simulationen Verlet-Integration oder Runge-Kutta besser wären, aber für dieses Projekt ist der Einfachheit halber Euler völlig okay.

Minkowski-basierte Kollision (Tag 050)

Dies ist einer der mathematisch interessantesten Tage. Statt Kollision zwischen zwei Rechtecken zu testen, verwendet Casey das Minkowski-Differenz-Prinzip. Das ist ein elegantes geometrisches Konzept, das ein 2-Body-Problem in ein 1-Body-Problem verwandelt.

🔷 Minkowski-Summe vereinfacht

Statt "Rechteck A kollidiert mit Rechteck B?" fragen wir: "Liegt der Mittelpunkt von A innerhalb von B, das um die halbe Größe von A in alle Richtungen erweitert wurde?" Das erweiterte Rechteck nennt man die Minkowski-Summe von B und dem Spiegelbild von A.

Mathematisch: Wenn A die halbe Breite wA und halbe Höhe hA hat, und B die halbe Breite wB und halbe Höhe hB, dann ist das erweiterte Rechteck ein Rechteck mit halber Breite wA + wB und halber Höhe hA + hB, zentriert an der Position von B.

Das ermöglicht nicht nur die Erkennung ob eine Kollision stattfindet, sondern auch die Berechnung der minimalen Überschneidungstiefe in jeder Richtung. Wenn der Mittelpunkt von A im erweiterten Rechteck liegt, können wir berechnen, wie viel er in X-Richtung und wie viel er in Y-Richtung zu weit eingedrungen ist. Das ist essenziell für saubere Kollisionsauflösung.

Minkowski-Kollisionsauflösung
// Entity A (Spieler) und Entity B (Wand)
real32 wA = 0.5f * A.Width;
real32 hA = 0.5f * A.Height;
real32 wB = 0.5f * B.Width;
real32 hB = 0.5f * B.Height;

// Erweitertes Rechteck (Minkowski-Summe)
real32 ExtW = wA + wB;
real32 ExtH = hA + hB;

// Abstand der Mittelpunkte
real32 dX = A.P.x - B.P.x;
real32 dY = A.P.y - B.P.y;

// Wenn der Abstand kleiner als die erweiterte Hälfte ist, gibt es eine Kollision
if(AbsoluteValue(dX) < ExtW && AbsoluteValue(dY) < ExtH)
{
    // Wie tief ist die Überschneidung?
    real32 OverlapX = ExtW - AbsoluteValue(dX);
    real32 OverlapY = ExtH - AbsoluteValue(dY);
    
    // Wir schieben in die Richtung mit der geringeren Überschneidung
    if(OverlapX < OverlapY)
    {
        if(dX > 0) A.P.x += OverlapX;
        else A.P.x -= OverlapX;
    }
    else
    {
        if(dY > 0) A.P.y += OverlapY;
        else A.P.y -= OverlapY;
    }
}

Was ist eine Race Condition?

Eine Race Condition tritt auf, wenn zwei oder mehr Threads gleichzeitig auf dieselben Daten zugreifen und mindestens einer von ihnen schreibt. Das Ergebnis hängt dann von der zufälligen Ausführungsreihenfolge ab — es ist nicht deterministisch. Das ist einer der schwierigsten Bug-Typen in der Software-Entwicklung, weil er sich oft nur unter Last zeigt und schwer reproduzierbar ist.

Schritt 10: Einen Worker-Thread erstellen

Bevor du komplexe Thread-Pools baust, musst du verstehen, wie ein einzelner Thread funktioniert. Unter Windows erstellst du einen Thread mit CreateThread. Der Thread führt eine Funktion aus, die du ihm übergibst.

Einfacher Worker-Thread
DWORD WINAPI
ThreadProc(LPVOID Parameter)
{
    int ThreadIndex = *(int *)Parameter;
    
    for(int i = 0; i < 10; ++i)
    {
        printf("Thread %d: Iteration %d\n", ThreadIndex, i);
    }
    
    return 0;
}

// Erstelle 4 Threads
for(int ThreadIndex = 0; ThreadIndex < 4; ++ThreadIndex)
{
    DWORD ThreadID;
    HANDLE ThreadHandle = CreateThread(0, 0, ThreadProc, &ThreadIndex, 0, &ThreadID);
}

Wichtiger Hinweis: Im obigen Code gibt es einen subtilen Bug! Wir übergeben den Pointer auf ThreadIndex, aber die äußere Schleife ändert ThreadIndex sofort, während der Thread noch startet. Das ist eine klassische Race Condition. Die Lösung: Entweder auf das Starten des Threads warten (WaitForSingleObject) oder für jeden Thread einen separaten Speicherbereich allokieren.

Interlocked Operations

In Tag 123 werden atomic operations eingeführt — speziell die Windows-Funktionen InterlockedIncrement, InterlockedCompareExchange und InterlockedExchangeAdd. Diese Operationen werden direkt von der CPU in einem einzigen, unteilbaren Schritt ausgeführt. Sie sind die Bausteine für alle lock-freien Datenstrukturen.

Warum "interlocked"? Der Name kommt daher, dass die CPU sicherstellt, dass kein anderer Kern gleichzeitig auf dieselbe Speicherstelle zugreifen kann. Auf x86-CPUs geschieht das über den LOCK-Präfix vor der Instruktion, der den Speicherbus für die Dauer der Operation sperrt.

Lock-freie Queue (vereinfacht)
struct work_queue_entry
{
    char *StringToPrint;
};

struct work_queue
{
    work_queue_entry Entries[256];
    uint32 volatile CompletionGoal;
    uint32 volatile CompletionCount;
    
    uint32 volatile NextEntryToWrite;
    uint32 volatile NextEntryToRead;
};

bool32 DoNextWorkQueueEntry(work_queue *Queue)
{
    uint32 OriginalNextEntryToRead = Queue->NextEntryToRead;
    uint32 NewNextEntryToRead = (OriginalNextEntryToRead + 1) % ArrayCount(Queue->Entries);
    
    if(OriginalNextEntryToRead != Queue->NextEntryToWrite)
    {
        uint32 Index = InterlockedCompareExchange(
            (LONG volatile *)&Queue->NextEntryToRead,
            NewNextEntryToRead,
            OriginalNextEntryToRead);
            
        if(Index == OriginalNextEntryToRead)
        {
            // Wir haben erfolgreich einen Job reserviert
            work_queue_entry Entry = Queue->Entries[Index];
            Entry.Callback(Queue, Entry.Data);
            InterlockedIncrement((LONG volatile *)&Queue->CompletionCount);
            return true;
        }
    }
    return false;
}

Diese Queue ist lock-frei (lock-free): Kein Thread muss jemals auf ein Mutex warten. Stattdessen verwenden wir InterlockedCompareExchange (CAS), um atomar zu prüfen, ob der aktuelle Lese-Index noch unserem erwarteten Wert entspricht, und ihn dann zu inkrementieren. Wenn zwei Threads gleichzeitig zugreifen, "verliert" einer den Vergleich und versucht es erneut.

Warum ist das wichtig? Mutexe (oder Critical Sections) sind langsam, weil sie den Thread in den Kernel-Modus versetzen können, wenn er blockiert. Bei einer lock-freien Queue "spinnt" der Thread nur ein paar Nanosekunden in einer Schleife, bevor er weiterarbeitet. Das ist um Größenordnungen schneller, besonders wenn viele Threads auf die Queue zugreifen.

⚠️ Memory Barriers

In Tag 124 erklärt Casey das kritische Konzept der Memory Barriers (oder Memory Fences). Moderne CPUs und Compiler dürfen Instruktionen umordnen, wenn sie keine Datenabhängigkeit erkennen. Bei Multithreading kann das katastrophal sein.

Stell dir vor, Thread A schreibt Daten in einen Job und inkrementiert dann NextEntryToWrite. Thread B liest NextEntryToWrite und greift auf die Daten zu. Wenn der Compiler oder die CPU die Schreiboperationen umordnet, könnte Thread B den neuen NextEntryToWrite sehen, bevor die eigentlichen Daten geschrieben wurden. Das Ergebnis: Thread B liest korrupte oder alte Daten.

Auf Windows sorgen die Interlocked*-Funktionen implizit für Memory Barriers. Aber wenn du eigene lock-freie Strukturen baust, musst du explizit Barrieren einfügen (z. B. _ReadWriteBarrier auf Windows oder std::atomic in C++).

Tag 115–120: SIMD-Basics

SIMD steht für Single Instruction, Multiple Data. Moderne CPUs haben spezielle Register (z. B. 128-Bit XMM-Register bei x86, 256-Bit YMM-Register bei AVX), die mehrere Werte gleichzeitig halten können. Eine einzige Instruktion kann dann z. B. vier 32-Bit-Floats gleichzeitig addieren, multiplizieren oder vergleichen.

Warum SIMD?

In der Computergrafik führst du oft dieselbe Operation auf vielen Pixeln gleichzeitig aus. Wenn du vier Pixel gleichzeitig verarbeiten kannst, steigerst du den Durchsatz theoretisch um den Faktor 4. In der Praxis ist der Faktor oft etwas niedriger, weil SIMD-Code mehr Overhead hat (Daten müssen aligned sein, Schleifen müssen für Rest-Elemente behandelt werden, etc.), aber für innere Schleifen kann SIMD enorme Unterschiede machen.

SSE-Beispiel: Vier Floats addieren
#include <intrin.h>

// __m128 ist ein 128-Bit-Register, das 4 floats hält
__m128 A = _mm_set_ps(1.0f, 2.0f, 3.0f, 4.0f);
__m128 B = _mm_set_ps(5.0f, 6.0f, 7.0f, 8.0f);
__m128 Result = _mm_add_ps(A, B);

// Result enthält jetzt: [6.0, 8.0, 10.0, 12.0]
// Beachte: _mm_set_ps nimmt die Argumente in REVERSE order!

Casey wendet SIMD zunächst auf den Software-Renderer an. Statt Pixel für Pixel zu verarbeiten, werden vier Pixel gleichzeitig geladen, entpackt (von 8-Bit-Integer zu 32-Bit-Float), mit Alpha-Werten multipliziert, addiert und wieder zu 8-Bit-Integer gepackt. Das kann den Renderer um den Faktor 3–4 beschleunigen.

Schritt 11: SIMD im Renderer verwenden

Wenn du Alpha-Blending für vier Pixel gleichzeitig machen willst, musst du:

  1. Vier 32-Bit-Pixel aus dem Speicher laden (_mm_loadu_si128)
  2. Sie in vier 32-Bit-Integer-Kanäle aufteilen (Shuffle-Operationen)
  3. Von Integer zu Float konvertieren (_mm_cvtepi32_ps)
  4. Die Alpha-Blending-Mathematik durchführen
  5. Zurück zu Integer konvertieren und packen
  6. In den Speicher schreiben

Das ist viel Code, aber der Geschwindigkeitsvorteil lohnt sich für innere Schleifen.

Tag 112–114: Performance-Analyse

Bevor man optimiert, muss man messen. Casey erklärt in diesen Tagen die Grundlagen der CPU-Performance: Pipelines, Cache-Hierarchien, Branch Prediction und Out-of-Order-Execution.

Die Cache-Hierarchie

Moderne CPUs haben mehrere Ebenen von Caches:

  • L1-Cache: ~32 KB, extrem schnell (4 Zyklen Latenz), pro Kern
  • L2-Cache: ~256 KB, sehr schnell (12 Zyklen), pro Kern
  • L3-Cache: ~8–32 MB, schnell (40 Zyklen), geteilt zwischen Kernen
  • Hauptspeicher (RAM): Gigabytes, langsam (200+ Zyklen)

Wenn dein Algorithmus Daten aus dem RAM holen muss, wartet die CPU Hunderte von Zyklen. Wenn die Daten im L1-Cache sind, ist der Zugriff praktisch instant. Deshalb ist Cache-Freundlichkeit oft wichtiger als die algorithmische Komplexität. Ein O(n²)-Algorithmus mit gutem Cache-Verhalten kann schneller sein als ein O(n log n)-Algorithmus, der ständig im RAM herumspringt.

Branch Prediction

Moderne CPUs führen Instruktionen spekulativ aus. Wenn sie auf eine if-Anweisung stoßen, raten sie, welcher Zweig wahrscheinlich genommen wird, und führen diesen schon aus, bevor das Ergebnis der Bedingung feststeht. Wenn die CPU richtig geraten hat, spart das enorm Zeit. Wenn sie falsch geraten hat (Branch Misprediction), muss sie alle spekulativ ausgeführten Instruktionen verwerfen — ein teurer Vorgang.

Praktische Konsequenz: Wenn du in einer heißen Schleife eine Bedingung hast, die zufällig wahr oder falsch ist (z. B. if (IsColliding) bei gleichmäßig verteilten Objekten), wird die CPU oft falsch raten. Wenn möglich, solltest du solche Bedingungen vermeiden oder die Daten so sortieren, dass die Bedingung vorhersehbar ist (z. B. alle kollidierenden Objekte zusammenhalten).

Der Mixer-Loop

Für jeden Output-Sample müssen wir alle aktiven Sound-Quellen summieren. Das klingt einfach, wird aber komplex, wenn Sounds unterschiedliche Sample-Raten haben oder ihre Lautstärke über die Zeit verändert wird.

Mixer-Grundprinzip
for(int SampleIndex = 0; SampleIndex < SampleCount; ++SampleIndex)
{
    int32 SampleValue = 0;
    
    playing_sound **SoundPtr = &Mixer->FirstPlayingSound;
    while(*SoundPtr)
    {
        playing_sound *Sound = *SoundPtr;
        
        // Lies den aktuellen Sample-Wert aus dem Source-Buffer
        real32 AudioValue = GetNextAudioSample(Sound);
        
        // Interpoliere Lautstärke (für Fade-In/Out)
        Sound->CurrentVolume += Sound->dVolume;
        
        SampleValue += (int32)(AudioValue * Sound->CurrentVolume);
        
        // Wenn der Sound zu Ende ist, entferne ihn aus der Liste
        if(Sound->Finished)
        {
            *SoundPtr = Sound->Next;
        }
        else
        {
            SoundPtr = &Sound->Next;
        }
    }
    
    *Dest++ = SampleValue;
}

Beachte die Linked-List-Traversierung mit einem Double-Pointer (SoundPtr): Das ist ein eleganter C-Trick, um Elemente während der Iteration zu entfernen, ohne einen separaten "Previous"-Pointer zu brauchen. Wenn der Sound fertig ist, schreiben wir einfach Sound->Next in den Pointer, der auf den aktuellen Sound zeigt — egal ob es der Kopf der Liste oder ein Next-Feld eines vorherigen Elements ist.

Pitch-Shifting und Resampling

In Tag 143 wird Pitch-Shifting implementiert. Wenn ein Sound schneller oder langsamer abgespielt werden soll (z. B. Schritte mit unterschiedlicher Geschwindigkeit oder ein Motorengeräusch, das sich mit der Drehzahl ändert), müssen wir nicht bei jedem Output-Sample einen neuen Wert aus dem Source-Buffer lesen. Stattdessen laufen wir mit einem fraktionalen Index durch den Source-Buffer.

Resampling mit fraktionalem Index
// Sound->SampleIndex ist ein real32, nicht ein int!
int32 SampleIndex0 = (int32)Sound->SampleIndex;
int32 SampleIndex1 = SampleIndex0 + 1;
real32 t = Sound->SampleIndex - (real32)SampleIndex0; // Fraktionaler Teil

// Lineare Interpolation zwischen zwei benachbarten Samples
real32 Sample0 = (real32)Sound->Samples[SampleIndex0];
real32 Sample1 = (real32)Sound->Samples[SampleIndex1];
real32 AudioValue = (1.0f - t) * Sample0 + t * Sample1;

// Advance um die Sample-Rate-Verhältnis
Sound->SampleIndex += (Sound->SourceSampleRate / (real32)OutputSampleRate);

Die lineare Interpolation (lerp) verhindert, dass der Sound "kratzt" oder verzerrt, wenn die Abspielgeschwindigkeit nicht exakt 1:1 zur Sample-Rate ist. Für höhere Qualität könnte man stattdessen kubische Interpolation verwenden, aber für Spiele ist lineare Interpolation meist ausreichend.

Tag 147–155: Asset-System & HHA-Format

📦 Warum ein eigenes Asset-Format?

Standardformate wie PNG oder WAV sind für den Laufzeit-Import zu langsam und komplex. Ein PNG muss dekomprimiert werden (ZLIB/DEFLATE), ein WAV hat einen Header, der geparst werden muss. Ein eigenes Binary-Format kann exakt auf die Bedürfnisse der Engine zugeschnitten werden:

  • Assets sind bereits in GPU-freundliche Formate konvertiert
  • Sie sind mit Tags versehen für schnelle Suche
  • Sie sind für schnelles Streaming optimiert (keine Dekompression nötig)
  • Der Header ist minimal und schnell zu parsen

Das HHA-Format hat einen Header gefolgt von Typed Asset Arrays. Jedes Asset hat einen Typ (Bitmap, Sound, Font), eine ID und optionale Tags (z. B. "Wald", "Tag", "Gegner"). Das ermöglicht später eine effiziente Suche und Filterung.

Die Anatomie einer Schriftart

Um Text professionell zu rendern, musst du mehrere metrische Werte verstehen:

  • Baseline: Die imaginäre Linie, auf der die meisten Buchstaben "stehen"
  • Ascent: Der Abstand von der Baseline zur höchsten Stelle (z. B. bei "b" oder "l")
  • Descent: Der Abstand von der Baseline zur tiefsten Stelle (z. B. bei "g" oder "p")
  • Line Gap: Der zusätzliche Abstand zwischen zwei Zeilen
  • Kerning: Der spezifische Abstand zwischen bestimmten Buchstabenpaaren (z. B. "AV")

💡 Warum Kerning wichtig ist

Ohne Kerning würde der Abstand zwischen allen Buchstaben gleich sein. Das sieht bei Buchstaben wie "A" und "V" extrem hässlich aus, weil der schräge Strich des "V" viel näher an "A" herankommen könnte. Ein professionelles Font-System berücksichtigt Kerning-Paare und verschiebt die Buchstaben entsprechend.

General Purpose Allocator (Tag 157–161)

Bisher hat das Spiel nur mit einfachen Memory Arenas gearbeitet (linearer Offset-Allocator). In dieser Phase wird ein vollständiger General Purpose Allocator entwickelt, der beliebige Allokationen und Deallokationen beliebiger Größe unterstützt.

Casey implementiert einen klassischen Free-List-basierten Allocator: Der Speicher wird in Blöcke unterschiedlicher Größe unterteilt. Freie Blöcke werden in Listen verwaltet. Bei einer Allokation wird der passendste freie Block gesucht und aufgeteilt, falls er zu groß ist.

Warum brauchen wir das? Arenas sind perfekt für temporären Speicher und für Daten, die nie freigegeben werden müssen. Aber manche Systeme brauchen dynamische Allokationen: Der Asset-Loader muss Speicher für verschieden große Texturen allokieren und später wieder freigeben. Der Debug-System muss Speicher für variabel große Datenblöcke verwalten. Für diese Fälle brauchen wir einen echten Allocator.

#define TIMED_BLOCK(Name) \
    timed_block TimedBlock_##Name(Name);
    
struct timed_block
{
    timed_block(char *Name)
    {
        StartTime = __rdtsc();
    }
    
    ~timed_block()
    {

// Verwendung — einfach am Anfang der Funktion:

}

Tag 229–234: Renderer Sorting

Wenn wir viele Sprites und Objekte haben, müssen wir sie in der richtigen Reihenfolge zeichnen — z. B. damit ein Charakter vor einem Baum steht, aber hinter einer Hauswand. Casey führt Sort Keys ein: Jedem Render-Element wird ein 64-Bit-Integer zugewiesen, der seine Zeichenreihenfolge kodiert.

Warum 64-Bit Sort Keys?

Ein 64-Bit-Integer erlaubt es uns, mehrere Kriterien in einem einzigen Wert zu kodieren. Wir können verschiedene Bit-Bereiche für verschiedene Eigenschaften reservieren:

  • Bits 48–63: Z-Layer (z. B. Hintergrund, Welt, Vordergrund, UI)
  • Bits 24–47: Y-Position (für isometrische oder top-down Sortierung)
  • Bits 8–23: Material-ID (für Batch-Rendering)
  • Bits 0–7: Flags

Wenn wir die Render-Elemente nach diesem Integer sortieren, erhalten wir automatisch die richtige Reihenfolge: Zuerst alle Hintergrund-Elemente, dann die Welt-Elemente sortiert nach Y-Position, dann die UI-Elemente.

Sort Key Struktur
union render_sort_key
{
    struct
    {
        uint64 ZLayer : 16;
        uint64 YPosition : 24;
        uint64 Material : 16;
        uint64 Flags : 8;
    };
    uint64 Value;
};

// Beispiel: Ein Baum im Hintergrund
render_sort_key TreeKey = {};
TreeKey.ZLayer = 1;      // Hintergrund-Layer
TreeKey.YPosition = 100; // Y-Position in der Welt
TreeKey.Material = 5;    // Baum-Texture

Radix Sort

In Tag 234 wird Radix Sort implementiert. Radix Sort ist ein nicht-vergleichender Sortieralgorithmus, der in O(n · k) arbeitet — wobei k die Anzahl der Zifferngruppen ist. Für 64-Bit-Keys sortiert man typischerweise in 8 Durchgängen (je 8 Bit) mit Counting Sort, was in der Praxis linear skaliert.

Warum Radix Sort statt Quicksort? Quicksort hat eine durchschnittliche Komplexität von O(n log n). Für 10.000 Render-Elemente bedeutet das etwa 10.000 × log₂(10.000) ≈ 130.000 Vergleiche — wobei jeder Vergleich eine Branch-Prediction- Herausforderung für die CPU ist. Radix Sort braucht 8 lineare Durchgänge ohne Verzweigungen. Das ist dank besserer Cache-Nutzung und fehlender Branch Mispredictions in der Praxis oft deutlich schneller.

💡 Wie funktioniert Radix Sort?

Radix Sort sortiert Zahlen nicht durch Vergleiche, sondern durch Betrachten einzelner Ziffern (oder Bits). Für 32-Bit-Integers sortiert man zuerst nach den unteren 8 Bits, dann nach den nächsten 8 Bits, dann nach den nächsten 8 Bits, und schließlich nach den oberen 8 Bits. Nach jedem Durchgang sind die Zahlen bezüglich der betrachteten Bits sortiert. Der Trick ist, dass jeder einzelne Durchgang mit Counting Sort in O(n) durchgeführt werden kann, weil es nur 256 mögliche Werte für 8 Bits gibt.

Sort Rules statt Sort Keys

In Tag 300 stellt Casey fest, dass totale Ordnung (ein einzelner Sort-Key pro Objekt) nicht ausreicht. Manche Objekte haben komplexe Überlappungsbeziehungen, die sich nicht in einem einfachen Key ausdrücken lassen. Was ist, wenn Objekt A vor B gezeichnet werden muss, B vor C, aber C vor A? Das ist ein Zyklus — es gibt keine totale Ordnung.

Casey führt daher Sort Rules und später Graph-based Sorting ein: Ein gerichteter Graph beschreibt, welches Objekt vor welchem anderen gezeichnet werden muss. Wenn der Graph zyklisch ist, muss man entweder einen Zyklus brechen (indem man eine Kante entfernt) oder die Objekte anders anordnen.

Tag 235–244: OpenGL-Initialisierung

OpenGL unter Windows ist nicht trivial. Casey erklärt das Problem des Pixel-Format-Deskriptors: Zuerst brauchen wir einen temporären OpenGL-Context, um die Erweiterungs-Funktion wglChoosePixelFormatARB laden zu können. Mit dieser können wir dann den "echten" modernen Context mit den gewünschten Attributen erstellen.

⚠️ Der OpenGL Bootstrap-Problem

Windows liefert nur einen veralteten OpenGL 1.1-Context out-of-the-box. Um moderne Features (Shader, VBOs, FBOs) zu nutzen, muss man über Erweiterungen gehen. Aber um Erweiterungen zu laden, braucht man bereits einen OpenGL-Context. Die Lösung: Einen temporären Dummy-Context erstellen, wglGetProcAddress nutzen, um die modernen Funktionen zu laden, dann den Dummy-Context zerstören und den echten erstellen.

Das ist einer der hässlichsten Teile der Windows-OpenGL-Programmierung, und fast jeder Anfänger stolpert hier mindestens einmal darüber.

Asynchronous Texture Downloads

In Tag 243–244 wird ein System für asynchrone Textur-Downloads gebaut. Die Idee: Der Haupt-Thread rendert das Spiel, während Worker-Threads neue Texturen in den GPU-Speicher laden. Das erfordert mehrere OpenGL-Contexts, die sich über Share Lists GPU-Ressourcen wie Texturen und Buffer teilen.

Warum ist das wichtig? Wenn du eine große Textur synchron lädst, blockiert der GPU-Thread möglicherweise für mehrere Millisekunden. Bei 60 FPS hast du nur 16.67ms pro Frame — ein einziger synchroner Texture-Download kann einen Frame Drop verursachen. Mit asynchronem Loading passiert das im Hintergrund.

Tag 368–372: Shader & Modern OpenGL

Casey wechselt vom Fixed-Function-Pipeline-OpenGL zum Core Profile. Das bedeutet: Keine fest eingebauten Transformationen mehr, keine festen Lichtquellen, keine immediate-mode Zeichenbefehle wie glBegin/glEnd. Stattdessen schreiben wir eigene Vertex Shader und Fragment Shader in GLSL.

Minimaler Vertex Shader
#version 330 core
layout(location = 0) in vec3 Position;
layout(location = 1) in vec2 UV;
layout(location = 2) in vec4 Color;

uniform mat4 Transform;
out vec2 FragUV;
out vec4 FragColor;

void main()
{
    gl_Position = Transform * vec4(Position, 1.0);
    FragUV = UV;
    FragColor = Color;
}

Der Vertex Shader läuft einmal pro Vertex und transformiert die Position vom lokalen Modellraum in den Clip-Space. Der Fragment Shader läuft einmal pro Pixel (genauer: einmal pro Fragment) und bestimmt die finale Farbe.

Vertex Arrays und Buffer Objects

Im modernen OpenGL speicherst du deine Geometrie in Vertex Buffer Objects (VBOs). Das sind Speicherbereiche auf der GPU. Du lädst deine Vertex-Daten einmalig in den VBO und sagt OpenGL dann nur noch "zeichne diese 1000 Dreiecke". Das ist um Größenordnungen schneller als immediate mode, bei dem du für jeden Vertex eine separate Funktion aufrufst.

Tag 361–367: 3D-Mathematik

Die Mathematik wird nun dreidimensional. Casey führt Rotationsmatrizen ein — zunächst für die einzelnen Achsen (X, Y, Z), dann für beliebige Rotationen.

Matrix-Multiplikation und Transformations-Reihenfolge

Wenn du einen 3D-Punkt transformieren willst, multiplizierst du ihn mit einer 4×4-Matrix. Aber die Reihenfolge ist entscheidend! Wenn du ein Objekt erst skalieren, dann rotieren und dann verschieben willst, musst du die Matrizen in umgekehrter Reihenfolge multiplizieren:

M = Translate × Rotate × Scale

Das liegt daran, dass Matrizen-Multiplikation von rechts nach links angewendet wird. Wenn du den Punkt p mit M multiplizierst, passiert zuerst die Scale, dann die Rotate, dann die Translate.

Perspektivische Projektion
// Einfache Perspective-Projektions-Matrix
inline m4x4 Perspective(real32 FOV, real32 Aspect, 
                         real32 Near, real32 Far)
{
    real32 TanHalfFOV = Tan(FOV * 0.5f);
    
    m4x4 Result = {};
    Result.E[0][0] = 1.0f / (Aspect * TanHalfFOV);
    Result.E[1][1] = 1.0f / TanHalfFOV;
    Result.E[2][2] = -(Far + Near) / (Far - Near);
    Result.E[2][3] = -1.0f;
    Result.E[3][2] = -(2.0f * Far * Near) / (Far - Near);
    
    return Result;
}

Diese Matrix führt zwei Dinge gleichzeitig aus: Sie skaliert die X- und Y-Koordinaten basierend auf dem Sichtfeld (FOV) und dem Seitenverhältnis, und sie transformiert die Z-Koordinate in den Bereich [-1, 1] für den Depth Buffer. Das -1 in E[2][3] ist der Schlüssel: Es kopiert die Welt-Z-Koordinate in die homogene W-Komponente. Später teilt OpenGL alle Koordinaten durch W — das ist die Perspective Division, die entfernte Objekte kleiner erscheinen lässt.

Depth Buffer

In Tag 364 wird der Depth Buffer aktiviert. OpenGL speichert für jeden Pixel die Tiefe (Z-Wert) des gerenderten Fragments. Wenn ein neues Fragment gezeichnet wird, wird es nur akzeptiert, wenn es näher an der Kamera ist. Das ermöglicht korrektes Überdecken von 3D-Objekten ohne manuelles Sortieren.

Wichtig: Der Depth Buffer hat eine begrenzte Präzision (meist 24 Bit). Wenn dein Near-Plane sehr nah und dein Far-Plane sehr weit ist, verlierst du Präzision in der Mitte. Das führt zu Z-Fighting — ein Flackern, wenn zwei Oberflächen fast dieselbe Tiefe haben. Casey wählt daher die Near- und Far-Planes so eng wie möglich.

Phase 09 Tag 401 – 500

Beleuchtung, Raycasting & Rendering

Das Herzstück der visuellen Qualität. Wir bauen ein komplexes Beleuchtungssystem mit Depth Peeling, Voxel-basiertem Raycasting und Octahedral Light Atlases.

Tag 381–389: Depth Peeling

Depth Peeling ist eine Technik, um transparente oder überlappende Objekte korrekt zu rendern. Der normale Depth Buffer kann nur ein Objekt pro Pixel speichern. Wenn du mehrere transparente Objekte übereinander hast (z. B. Glas vor Wasser vor Rauch), reicht das nicht aus.

🔷 Wie funktioniert Depth Peeling?

Pass 1: Rendere die Szene wie gewohnt mit Depth Test. Speichere das Ergebnis als "Layer 0". Pass 2: Setze den Depth Test so, dass er tiefer als Layer 0 liegende Fragmente akzeptiert, aber alles, was näher ist, verwirft. Das Ergebnis ist Layer 1. Wiederhole für so viele Layer wie nötig (z. B. 4–8).

Am Ende hast du 4–8 Texturen, die von vorne nach hinten sortiert sind. Du kannst sie von hinten nach vorne überblenden, um das korrekte transparente Bild zu erhalten.

In Tag 388 wird Depth Peeling mit Multisampling (MSAA) kombiniert. Das ist besonders knifflig, weil Multisampling mehrere Samples pro Pixel hat, und der Depth Test für jedes Sample separat durchgeführt werden muss. Casey implementiert einen Custom Multisample Resolve, der die Samples korrekt zu einem finalen Pixel zusammenfasst.

Tag 401–434: Voxel-basiertes Raycasting

Die Beleuchtung in Handmade Hero ist nicht einfach ein paar Punktlichter. Es ist ein globales Beleuchtungssystem, das simuliert, wie Licht durch die Szene gestreut und reflektiert wird. Casey verwendet dafür Voxel-Raycasting.

Monte-Carlo-Sampling

Die Welt wird in einen 3D-Voxel-Grid unterteilt. Für jeden Voxel wird berechnet, wie viel Licht von verschiedenen Richtungen einfällt. Das geschieht durch Monte-Carlo-Sampling: Vom Voxel aus werden Tausende von Strahlen (Rays) in zufällige Richtungen geschossen. Wenn ein Strahl ein Objekt trifft, wird das Licht dieses Objekts zum Voxel beigetragen.

Grundprinzip des Voxel-Raycasters
for(int RayIndex = 0; RayIndex < RayCount; ++RayIndex)
{
    v3 RayDirection = RandomDirection(RandomSeries);
    
    // Walk through the grid (DDA-Algorithmus)
    voxel_position VoxelP = StartVoxel;
    for(int Step = 0; Step < MaxSteps; ++Step)
    {
        if(IsOccupied(World, VoxelP))
        {
            light_value Light = GetLightAt(World, VoxelP);
            real32 CosTheta = Max(0.0f, Dot(RayDirection, Normal));
            AccumulatedLight += Light * CosTheta;
            break;
        }
        VoxelP = StepAlongRay(VoxelP, RayDirection);
    }
}

// Durchschnitt über alle Samples
AccumulatedLight /= RayCount;

Der CosTheta-Term ist das Lambert'sche Cosinus-Gesetz: Je steiler das Licht auf eine Oberfläche trifft, desto weniger Energie wird pro Fläche übertragen. Eine Oberfläche, die parallel zum Licht steht, empfängt die volle Intensität. Eine Oberfläche, die im 90-Grad-Winkel zum Licht steht, empfängt gar kein Licht.

Optimierung des Raycasters

Dieser naive Ansatz ist extrem langsam. In Tag 590–617 wird der Raycaster massiv optimiert:

  • SIMD: Mehrere Strahlen gleichzeitig verarbeiten
  • Grid Walk Table: Die Schritt-Berechnung vorberechnen
  • Early Termination: Strahlen abbrechen, die in leeren Bereichen verschwinden
  • Kd-Tree / Spatial Hierarchy: Leere Bereiche schnell überspringen

Tag 547–589: Light Probes & Octahedral Maps

Statt für jeden Pixel Licht zu berechnen, verwendet Casey Light Probes — diskrete Punkte in der Welt, an denen die Lichtverteilung gesampelt wird. Die Ergebnisse werden in Octahedral Atlases gespeichert.

💡 Octahedral Encoding

Eine Kugeloberfläche (alle möglichen Lichtrichtungen) lässt sich nicht effizient als 2D-Textur speichern. Das Octahedral Mapping projiziert die Kugel auf ein Quadrat, ähnlich wie eine Weltkarte die Erde projiziert. Das ermöglicht es, für jede Lichtrichtung einen UV-Koordinaten-Paar zu haben und die Lichtverteilung als normale 2D-Textur zu speichern.

Der Name kommt daher, dass die Projektion ein Oktaeder (eine Doppelpyramide mit 8 dreieckigen Seitenflächen) verwendet. Das Oktaeder wird dann in ein Quadrat abgeflacht.

Das System unterstützt sogar indirect diffuse lighting (Tag 584): Licht, das nicht direkt von einer Quelle kommt, sondern von anderen Oberflächen reflektiert wurde. Das wird iterativ berechnet: In jedem Schritt wird das Licht der vorherigen Iteration als Input für die nächste verwendet, bis es konvergiert. Das ist eine vereinfachte Form von Radiosity oder Path Tracing.

Tag 446–502: Prozedurale Weltgenerierung

Statt statische Level zu designen, generiert Handmade Hero seine Welten prozedural. Casey beginnt mit der Generierung von Room Volumes — rechteckigen Bereichen, die Räume darstellen. Diese Räume werden zufällig in der Welt platziert und dann durch Gänge verbunden.

Der Room Placement Algorithmus

Der Algorithmus arbeitet in mehreren Phasen:

  1. Seed-Room: Ein Start-Raum wird an einer festen Position platziert.
  2. Expansion: Neue Räume werden an bestehende Räume angebaut. Für jede Wand eines existierenden Raums wird geprüft, ob dort ein neuer Raum passen würde.
  3. Overlap Prevention: Wenn ein neuer Raum mit einem bestehenden kollidiert, wird er verworfen und ein anderer Versuch unternommen.
  4. Connection Validation: Jeder Raum muss von jedem anderen Raum aus erreichbar sein. Das wird durch einen Graphen-Algorithmus verifiziert.
Room-Overlap-Test
bool32 RoomsOverlap(room A, room B)
{
    bool32 Result = !((B.Max.x <= A.Min.x) ||
                     (B.Min.x >= A.Max.x) ||
                     (B.Max.y <= A.Min.y) ||
                     (B.Min.y >= A.Max.y));
    return Result;
}

Dieser Test ist die 2D-Version des Separating Axis Theorem (SAT). Zwei Rechtecke überlappen sich nicht, wenn es mindestens eine Achse gibt, auf der sie getrennt sind. Wenn sie sich auf der X-Achse und auf der Y-Achse überlappen, überlappen sie sich im 2D-Raum.

Undo / Redo System

In Tag 489 wird ein klassisches Command-Pattern-basiertes Undo/Redo-System implementiert. Jede Editor-Aktion (Move, Rotate, Scale, Delete) wird als Objekt gespeichert, das zwei Methoden hat: Execute und Undo. Die Aktionen werden in einem Stack verwaltet.

Command Pattern für Undo
struct edit_operation
{
    edit_operation *Next;
    edit_operation *Prev;
    
    void (*Apply)(edit_operation *Op, world *World);
    void (*Reverse)(edit_operation *Op, world *World);
    
    // Operation-spezifische Daten
    entity_id EntityID;
    v3 OriginalP;
    v3 NewP;
};

struct edit_history
{
    edit_operation *LastOperation;
};

void Undo(edit_history *History, world *World)
{
    if(History->LastOperation)
    {
        edit_operation *Op = History->LastOperation;
        Op->Reverse(Op, World);
        History->LastOperation = Op->Prev;
    }
}

Das Command Pattern ist elegant, weil es die Aktion von ihrer Ausführung trennt. Du kannst Aktionen nicht nur ausführen und rückgängig machen, sondern auch serialisieren (für Save/Load), wiederholen (für Makros), oder in Batches zusammenfassen (für komplexe Multi-Entity-Operationen).

Diese Punkte haben Namen und Transformationen (Position, Rotation, Skalierung relativ zur Entity).

Wenn sich eine Entity bewegt, wird ihre Bewegung in kleine Schritte unterteilt, und für jeden Schritt wird geprüft, ob der Ziel-Voxel begehbar ist.

🔷 Voxel Flood Fill für Kollision

In Tag 641 wird ein Flood-Fill-Algorithmus verwendet, um zu bestimmen, welche Voxel von einer Entity erreicht werden können. Wenn eine Entity in einer Wand "steckt" (z. B. durch einen Bug oder durch Teleportation), sucht der Algorithmus den nächstgelegenen freien Voxel und schiebt die Entity dorthin. Das verhindert, dass der Spieler jemals dauerhaft in der Geometrie gefangen ist.

Move Queues

In Tag 656–657 werden Move Queues eingeführt. Anstatt dass eine Entity sofort ihre Position ändert, wenn der Spieler eine Taste drückt, wird eine Bewegungsanforderung in eine Warteschlange gestellt. Das System verarbeitet diese Anforderungen Schritt für Schritt und kann dabei Kollisionen, Animationen und Übergänge berücksichtigen.

Warum Move Queues? Sie führen zu einem viel flüssigeren, "taktileren" Bewegungsgefühl. Wenn der Spieler "links, links, oben" drückt, werden diese Bewegungen nacheinander ausgeführt, anstatt dass der Charakter sofort zur Endposition teleportiert. Das ermöglicht auch komplexere Bewegungsmuster wie Schieben von Kisten oder Klettern.

Tag 590–620: SIMD-Optimierung des Raycasters

Der Voxel-Raycaster ist der performance-kritischste Teil der Engine, weil er Tausende von Strahlen pro Frame verarbeiten muss. Casey optimiert ihn mit mehreren Techniken:

  • SIMD-Ray-Packet-Tracing: Vier Strahlen werden gleichzeitig verarbeitet. Alle Berechnungen (DDA-Grid-Walk, AABB-Tests, Normale-Berechnung) laufen auf 128-Bit-Registern.
  • Tableless Grid Walk: Die Schritt-Richtung durch das Grid wird nicht mehr pro Strahl berechnet, sondern durch direkte Bit-Manipulationen bestimmt.
  • Early Termination: Wenn ein Strahl in einen leeren Bereich gerät, in dem er garantiert nichts mehr trifft, wird er sofort abgebrochen.
  • Kd-Tree für statische Geometrie: Für sehr große Szenen wird ein Beschleunigungsstruktur verwendet, um leere Bereiche schnell zu überspringen.
SIMD AABB-Intersection (vereinfacht)
// Teste 4 Strahlen gleichzeitig gegen einen AABB
__m128 tMin = _mm_set1_ps(-FLT_MAX);
__m128 tMax = _mm_set1_ps(FLT_MAX);

// Für jede Achse (X, Y, Z):
__m128 t1 = _mm_mul_ps(_mm_sub_ps(BoxMinX, RayOriginX), InvRayDirX);
__m128 t2 = _mm_mul_ps(_mm_sub_ps(BoxMaxX, RayOriginX), InvRayDirX);

__m128 tNear = _mm_min_ps(t1, t2);
__m128 tFar = _mm_max_ps(t1, t2);

tMin = _mm_max_ps(tMin, tNear);
tMax = _mm_min_ps(tMax, tFar);

__m128 HitMask = _mm_cmple_ps(tMin, tMax);
// HitMask enthält jetzt für jeden der 4 Strahlen, ob ein Treffer vorliegt

Beachte, wie elegant SIMD hier ist: Statt vier separaten AABB-Tests durchzuführen, führen wir einen einzigen Test durch, der vier Strahlen gleichzeitig behandelt. Die Operationen _mm_min_ps, _mm_max_ps und _mm_cmple_ps sind alle Single-Instruction-Multiple-Data — sie kosten nicht mehr als ihre skalaren Gegenstücke, aber verarbeiten vier Mal so viele Daten.

Die finale Architektur

Bis Tag 667 hat Handmade Hero eine bemerkenswerte Architektur erreicht:

  • Eine saubere Trennung zwischen Plattform- und Spiel-Code

💡 Das wichtigste Lernziel

Handmade Hero beweist, dass man eine professionelle Game-Engine komplett "from scratch" bauen kann, wenn man bereit ist, die Zeit zu investieren, jedes Problem wirklich zu verstehen. Das Wissen, das du in dieser Serie aufbaust, ist universell einsetzbar — egal ob du später C++, Rust, Zig oder eine andere Sprache verwendest.

Casey hat nicht nur eine Engine gebaut. Er hat eine Denkweise demonstriert: Frage Abstraktionen. Verstehe die Grundlagen. Messe, bevor du optimierst. Baue Tools, die deinen Workflow beschleunigen. Das ist das wahre Erbe von Handmade Hero.
